# Default brand logos

This folder ships the bundled fallback logos that `MailComposerService` renders
when an admin has not configured a `brand.logo_url` setting. They are also
referenced by the customer portal layout as a placeholder until the admin
uploads a real logo.

## Files

| File | Dimensions | Format | Used where |
|---|---|---|---|
| `tesserabx-logo-default.png` | 240 x 60 | Transparent PNG | Default header logo in branded HTML emails |
| `tesserabx-icon-default.png` | 256 x 256 | Square PNG | Favicon, avatar fallback, social-card preview |

Both files are simple, on-brand placeholders generated by
[`tasks/GenerateDefaultLogos.cfc`](../../../tasks/GenerateDefaultLogos.cfc).
They are intentionally minimal: a wordmark or letterform in the Bootstrap
primary blue. Replace them with your own assets before going to production
(see the "Replacing the defaults" section below).

## Constraints to keep when replacing

The email composer assumes:

- The **header logo** is roughly **240 x 60** (max width about 300 to keep
  mobile clients from cropping inside the 600px column). Transparent PNG so
  it sits on top of the configured `brand.primary_color` band cleanly.
- The **square icon** is **256 x 256**, a single solid background plus
  high-contrast foreground glyph, with enough padding from the edges to
  survive being rendered inside a circle (favicons, avatars).
- File size is small: under 20 KB for the wide logo, under 100 KB for the
  icon. Email clients fetch the logo on every open; keeping it small helps
  load times and image-blocker negotiation.

## Replacing the defaults

Two paths:

1. **Configure `brand.logo_url` in admin**
   (`/agent/admin/settings`). When a non-blank URL is set, the composer
   ignores the bundled file entirely and references the configured URL in
   the rendered HTML's `<img src>`. This is the recommended path because no
   container rebuild is required.

2. **Drop a replacement file at this path**. If you want a different
   default for unconfigured deployments (e.g., your fork of TesseraBX),
   replace `tesserabx-logo-default.png` and `tesserabx-icon-default.png` in
   place. Keep the file names so existing layout references continue to
   resolve.

## Where these are referenced

| Reference | Source |
|---|---|
| HTML email `<img src>` (when `brand.logo_url` is blank) | [`modules_app/core/models/MailComposerService.bx`](../../../modules_app/core/models/MailComposerService.bx) `resolveLogoTarget()` |
| Browser favicon fallback | future work; not wired in v1 |
| Avatar fallback | future work; the `AvatarHelper` currently renders a colored-initials badge instead |

## Regenerating

If you want to refresh the bundled placeholders (e.g., after changing the
primary brand color baked into the generator), re-run the task inside the
dev container:

```bash
docker compose -f compose.yaml -f compose.dev.yaml -f compose.override.yaml \
  exec app box task run tasks/GenerateDefaultLogos
```

The task overwrites both PNGs in place.

## Future work

- Retina (`@2x`) variants for high-DPI clients. The
  [`<picture>` element](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/picture)
  with `srcset` works in modern email clients (Apple Mail, iOS Mail) and
  degrades cleanly elsewhere.
- Dark-mode variants honored by `@media (prefers-color-scheme: dark)` in
  the email layout's single `<style>` block (Apple Mail supports it,
  Outlook ignores it, no client breaks).
- SVG versions for the customer portal favicon and any non-email contexts.
